Workload Management

The purpose of workload management is to limit the number of active queries in the system at any given time in order to avoid exhausting system resources such as memory, CPU, and disk I/O. This is accomplished in Pivotal Greenplum Database with role-based resource queues. A resource queue has attributes that limit the size and/or total number of queries that can be executed by the users (or roles) in that queue. Also, you can assign a priority level that controls the relative share of available CPU used by queries associated with the resource queue. By assigning all database roles to the appropriate resource queue, administrators can control concurrent user queries and prevent the system from being overloaded.

See also Manage Resource Queues.

Any configuration changes you make here require a restart of the database.

Workload Management

  • Max resource queues: Sets the maximum number of resource queues that can be created in a Greenplum Database system. Default = 8
  • Max resource portals per transaction: Sets the maximum number of simultaneously open user-declared cursors allowed per transaction. Note that an open cursor will hold an active query slot in a resource queue. Default = 64.
  • Resource select only: If set to ON, then SELECT, SELECT INTO, CREATE TABLE AS SELECT, and DECLARE CURSOR commands are evaluated. If set to off INSERT, UPDATE, and DELETE commands will be evaluated as well. Default:
  • Resource cleanup gangs on wait: If enabled, cleans up idle segment worker processes before taking a slot in the resource queue. Default:
  • State collection on resource queue usage: If enabled, collects statistics on resource queue usage. Default:

Query Prioritization (optional)

  • Enable query prioritization: Enables the query prioritization feature. This is enabled by default.
  • Sweeper interval (ms): Sets the interval at which CPU usage is recalculated for all active statements. The default value for this parameter should be sufficient for typical database operations. Default: 1000 ms.
  • CPU units per segment: Specifies the number of CPU cores per segment. The default is 4 for segments and 24 for the master, the correct values for the EMC Greenplum Data Computing Appliance. Each host checks its own postgresql.conf file for the value of this parameter.
